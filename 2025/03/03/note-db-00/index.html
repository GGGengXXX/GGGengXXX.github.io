数据库系统特点

  • 数据结构化
  • 数据共享
    • 同一部分数据为多用户共享
    • 文件系统
    • 数据库系统
  • 数据独立性
    • 物理独立性
      • 修改物理存储模式不影响 上层应用程序
    • 逻辑数据独立性
      • 修改逻辑模式 不影响 上层应用程序
  • 方便的用户接口
  • 统一的数据管理与控制功能
    • 完整性
      • 正确，有效，相容
      • 指和现实世界实际情况相符
        • 不能出现xxx年龄3000岁
    • 安全性
      • 主动安全
        • 有效防止数据被 非法访问 或者 非法修改
      • 被动安全
        • 数据被破坏后能 恢复
    • 并发控制
      • 多个用户能够同时 存取，修改 数据

模式结构（用户角度）

模式（schema）

姓名 性别 年龄 学号

用户使用数据库： 按照模式描述的结构，在模式 的 实例中存取数据

逻辑抽象：三级模式结构

  • 外模式
    • 也称为用户模式，为不同 用户群体 提供定制化的数据试图
  • 概念模式
    • 数据库的 全局逻辑结构，描述所有实体、属性、关系及约束。
    • 数据库设计的核心
  • 内模式
    • 物理存储结构
    • 对用户透明（看不见）优化存储效率与访问性能
  • 二级映像
    • 实现三个抽象层次的联系和转换

数据库体系结构

  • 集中式
  • 客户-服务器式
  • 分布式
  • 并行式

Explanation from Deepseek

1. 集中式数据库（像家里的大衣柜）

特点：所有东西都塞进一个柜子

  • 怎么用：数据、软件、硬件全放在一台电脑上（比如公司财务部唯一的一台服务器）。
  • 优点：
    • 管理简单，像整理衣柜一样方便（所有数据一目了然）
    • 适合小公司或数据量少的情况（衣服不多随便塞）
  • 缺点：
    • 柜子坏了全家没衣服穿（单点故障）
    • 衣服太多柜子塞不下（扩展性差）
  • 举例：早期银行柜台系统、小型超市收银系统

2. 客户-服务器式（像餐厅点餐）

特点：服务员和厨房分工合作

  • 怎么用：
    • 客户端：你用的手机/电脑（顾客点菜）
    • 服务端：后台数据库服务器（厨房做菜）
  • 优点：
    • 顾客不用进厨房（用户不碰底层数据）
    • 厨房专心炒菜（服务器专注数据处理）
  • 缺点：
    • 网络断了就吃不上饭（依赖网络）
    • 大排档接待不了旅行团（高并发吃力）
  • 举例：学校选课系统、医院挂号系统

3. 分布式数据库（像连锁超市）

特点：每个分店都有仓库

  • 怎么用：数据分散在不同地区的服务器（北京店存北方货，广州店存南方货）
  • 优点：
    • 分店货架空了不影响总店（容灾能力强）
    • 顾客就近买东西（访问速度快）
  • 缺点：
    • 调货需要时间（数据同步延迟）
    • 各分店记账要对得上（一致性难保证）
  • 举例：淘宝双十一、12306春运抢票

4. 并行式数据库（像超市收银台）

特点：多个收银员同时结账

  • 怎么用：多台服务器同时处理任务（5个收银台一起扫描商品）
  • 优点：
    • 结账队伍走得快（处理速度快）
    • 一个收银员病了其他人顶上（高可用性）
  • 缺点：
    • 需要统一收银系统（硬件配置要一致）
    • 顾客不能自己选收银台（任务调度复杂）
  • 举例：证券交易所实时交易、气象局天气预报

一句话总结区别

类型 核心特点 适合场景
集中式 一人干所有活 小数据量简单业务
客户服务器 前台点单后台做菜 普通企业应用
分布式 多地分仓协同工作 海量数据高并发访问
并行式 多人同时处理同类任务 实时计算复杂分析

日常类比：

  • 查成绩用客户服务器式（手机查分→学校服务器）
  • 刷抖音用分布式（视频分散在全国机房）
  • 银行转账用并行式（同时处理百万笔交易）
  • 个人记账本属于集中式（所有账单记在一个本子上）
