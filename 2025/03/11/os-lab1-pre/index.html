发现就是把一个 .S 和一个 .c 文件编译了一下

我们在 start.S 文件中会调用 init.c

而 init.c 会根据不同的实验做出不同的事情，在 lab1 中，我们只是调用了 printk 函数打印了一句话，当然 printk 是需要我们去实现的

心得体会

面对不同的文件，不能只看一个代码文件，而应该应用模块化的思维，以模块为单位进行理解。比如 printk 函数就涉及到许多不同的文件。machine.c 需要与硬件打交道，直接控制单个字符的读入和输出，而 printk.c 调用 machine.c 实现一串字符的输出，同时又将 outputk 作为指针传递给 print.c 由 print.c 负责解析字符串中的内容。

Exer 1.1

阅读 tools/readelf 目录下的 elf.h、 readelf.c 和 main.c 文件，并补全readelf.c 中缺少的代码。 readelf 函数需要输出 ELF 文件中所有节头中的地址信息，对于每个节头，输出格式为 “%d:0x%x\n”，其中的 %d 和 %x 分别代表序号和地址。正确完成 readelf.c 之后，在 tools/readelf 目录下执行 make 命令，即可生成可执行文件 readelf，它接受文件名作为参数，对 ELF 文件进行解析。可以执行 make hello生成测试用的 ELF 文件 hello，然后运行 ./readelf hello 来测试 readelf。

打开 elf.h 可以看到三个结构体，分别是 elf头，段头表，节头表

目标：读到每一个节的地址

elf头 读取 节头表的首地址(相对elf头的便宜) and 节头表中有多少项 and 每一项的大小

接下来循环 【节头表项数】次，每次由
$$
base+第i项 \times 节头表表项大小
$$

得到当前 节头表项，然后利用结构体访问 addr 字段即可。

Exer 1.3

通过查询 mmu.h 可以发现栈顶的地址为 0x80400000 同时，mmu.h 定义了栈顶的宏定义 KSTACKTOP 直接使用即可，防止硬编码

这里使用 j 而不是 jal 是因为 j 只是简单的跳转，jal 跳转之前会记录下一条指令的地址到 ra 因为内核启动过程中不需要返回之前的代码，所以不需要 jal ，设置 ra 反而可能干扰后续 ra 的使用f

Exer 1.4

machine.c 实现单个字符的读入和输出。

printk.c 中有一个 outputk() 函数，循环调用 machine.c 中的 printcharc 实现 多个字符 的输出

void outputk(void *data, const char *buf, size_t len)

printk.c 中的 printk() 函数实际上调用了 print.c 中的 vprintfmt()

vprintfmt() 接收 回调函数 outputk 进行打印输出字符串，待解析字符串 fmt 参数列表 ap

  1. 循环输出没有 % 的字符 遇到 % 或者 \0 则跳出循环
  2. 判断，如果遇到了终止符，则 返回
  3. 遇到 % 则开始解析
    1. 0 / - 判断左右对齐，填充类型
    2. 数字，表示占位的宽度
    3. 判断是否为 long long
  4. 不同的输出类型，对应 local help functions
