数据模型

这个过程就像将现实中的事物一步步翻译成计算机能理解的语言，每个阶段都在增加技术细节，同时逐渐远离人类的直觉。我们可以通过一个「图书馆管理系统」的例子来理解：

1. 现实世界（人类视角）

特征：完全人类可理解，但模糊不精确
例子：

  • 读者来图书馆借书
  • 书有书名、作者、分类
  • 管理员要记录谁借了哪本书，何时归还

问题：
“记录借书”只是一个模糊的需求，无法直接告诉计算机怎么做。

2. 概念模型（人类思维抽象）

特征：用人类语言描述核心概念和关系
工具：实体关系图（ER图）、流程图
例子：

  • 明确实体：读者、图书、借阅记录
  • 定义关系：
    • 一个读者可以借多本书
    • 一本书同一时间只能被一个读者借
    • 借阅记录包含借书时间、还书时间

关键作用：
将现实需求转化为「业务人员和技术人员都能看懂」的蓝图。

3. 逻辑模型（技术视角抽象）

特征：用技术语言描述结构，但未涉及具体技术
工具：关系型数据库设计、类图
例子：

  • 设计数据库表：

  • 定义约束：

    • 图书状态必须为“在库”才能被借出
    • 同一读者不能同时借超过5本书

人类理解难度↑：
需要理解主键、外键、数据约束等技术概念。

4. 物理模型（机器优化视角）

特征：针对具体技术实现做优化
工具：索引设计、存储引擎选择
例子：

  • 在 Book.ISBN 字段创建索引，加快按ISBN查询速度
  • 将借阅记录表拆分为两个分区：
    • 当前借阅（存储在SSD硬盘）
    • 历史记录（存储在机械硬盘）
  • 选择InnoDB存储引擎支持事务

人类理解难度↑↑：
需要了解存储结构、硬件特性、数据库底层机制。

5. 计算机世界（机器语言）

特征：二进制代码或数据库文件
例子：

  • MySQL的 .ibd 文件存储表数据
  • 硬盘上的二进制位：01001000 01100101 表示 “He”
  • 内存中的数据结构：B+树索引节点

人类理解难度↑↑↑：
完全脱离自然语言，只能通过工具（如数据库客户端）间接解读。

为什么人类理解能力减弱？

  1. 抽象层级变化：
    • 现实世界 → 概念模型：用人类语言描述
    • 计算机世界：用机器语言描述
  2. 技术细节增加：
    • 逻辑模型开始引入主键、外键
    • 物理模型涉及存储结构、硬件优化
  3. 直观性消失：
    • 最终存储的二进制数据，无法直接对应现实中的“书”或“读者”

类比：建造房屋

  • 现实世界：业主说“我要一栋现代风格的三层别墅”
  • 概念模型：建筑师画出房屋外观和房间布局草图
  • 逻辑模型：结构工程师设计梁柱承重方案
  • 物理模型：施工队选择钢筋型号和混凝土配比
  • 计算机世界：砖块、水泥、钢筋在工地上的实际堆砌

越是后面的阶段，业主（非专业人员）越难理解其中的技术细节。

概念模型

  • 概念模型用于信息世界的建模，是对现实世界的第一层抽象

数据库的概念模型是数据库设计的第一步，它用人类能理解的抽象方式描述现实世界中的业务需求和核心数据关系，不涉及任何技术细节（如表结构、数据类型）。它是业务人员和技术人员沟通的“桥梁”。

概念模型的3个核心特点

  1. 业务导向：关注“业务需要什么数据”，而非“数据库如何存储”
  2. 技术无关：不使用SQL、不定义字段类型，只描述实体和关系
  3. 可视化表达：常用ER图（实体关系图）呈现

概念模型的4个核心元素

元素 说明 示例（图书馆系统）
实体 独立存在的事物 读者、图书、管理员
属性 实体的特征 读者的姓名、图书的ISBN号
关系 实体间的交互 读者「借阅」图书
约束 业务规则限制 一个读者最多同时借5本书

ER数据模型

两种抽象手段

  • 分类

  • 聚集

1. 分类（Generalization/Specialization）

一句话总结：把事物从「大类别」细化到「小类别」，或者反过来合并同类项。

类比生活：

  • 分类就像整理衣柜：
    • 大类别（Generalization）：把「T恤、毛衣、外套」统一归类为「上衣」。
    • 小类别（Specialization）：把「上衣」细分成「长袖T恤、短袖T恤、卫衣」。

数据库中怎么用：

  • 向上抽象（Generalization）：把多个实体类型的共同属性提取成一个更高层的实体。
    • 比如：学生和老师都有「姓名、年龄」，可以抽象出一个「人」实体，学生和老师继承自「人」。
  • 向下细化（Specialization）：根据某些特性，将一个实体拆分成多个子类型。
    • 比如：把「用户」拆分为「普通用户」和「VIP用户」，VIP用户额外有「会员等级」属性。

举个栗子🌰：

  • 分类的优点是：减少重复属性，逻辑清晰，方便扩展新类型。

2. 聚集（Aggregation）

一句话总结：把多个实体和它们的关系「打包」成一个更高级的实体，像乐高积木组合。

类比生活：

  • 比如组装电脑：
    • 单独的「CPU、内存、硬盘」是零件（实体），通过「组装」（关系）形成一个「电脑主机」（聚集实体）。

数据库中怎么用：

  • 当一个关系本身需要被当作实体处理时，就用聚集。
    • 比如：学生选课的关系（学生-课程）本身可能有属性「选课时间」，此时可以将「学生+课程+选课时间」打包成一个「选课记录」实体。

举个栗子🌰：

  • 这里「订单」和「商品」通过「包含」关系组合，形成「订单项」（聚集实体），方便统计总价。

两者的核心区别

分类 聚集
处理层级关系（父子类） 处理组合关系（整体与部分）
纵向抽象（上下层） 横向抽象（平级合并）
目的是继承属性和行为 目的是简化复杂关系的操作

应用场景

  • 用分类：当需要区分「VIP用户和普通用户」，或者统一管理「员工、学生、老师」的共同属性。
  • 用聚集：当需要把「订单+商品+购买数量」当作一个整体处理，或者统计「项目+参与员工+工时」。

总结：分类是「上下级划分」，聚集是「打包组合」。就像整理房间时，分类是把衣服按季节归类，聚集是把床、床头柜、台灯组合成「睡眠区」！

三个基本元素

  • 实体

  • 属性

  • 联系

  • 一个实体可以由若干个属性来刻画

  • 属性类型相同的实体属于一个实体集

简单属性 && 复合属性：属性能否再分 （街道=家庭住址+门牌号

单值属性 && 多值属性 ：某个实体能否有多个取值（一个人可以有多个电话号码

键 key

  • 实体集中能唯一标识某个实体的属性
  • 多个属性构成键 -> 复合键

联系

  • 由一系列实体之间的对应关系组成

  • 可以包含非键属性

图形化的方法——ER图

圈⭕ -> 属性

方形 -> 实体

菱形 -> 联系

联系的元——参与联系的实体的个数

基数比约束

  • 表示一个实体能通过联系关联到的另一个实体集中的实体的数量
  • 二元联系的两个实体集实际上属于同一个实体集的两个副本 -> 一元联系

约束

  • 约束是 对数据库 的 断言
  • 断言内容： 对 数据 进行 限定
  • 约束 是 数据库模式的 重要组成 部分
    • 非空约束 唯一值约束 主键约束 外键约束 check约束

键约束

  • 引用实体

  • 超键 super key

    • 一个或一组属性 —— 唯一确定实体

  • 候选键 candidate key

    • 某个超键的最小集

  • 主键 primary key

    • 任一个候选键

e.g.

超键：（学号 身份证 年龄） （学号 性别 班级）

候选键： （学号） （身份证）

主键： （学号）

ER建模准则

  1. 忠实于用户需求
  2. KISS原则 (keep it simple and stupid)
  3. 避免冗余
  4. 能抽象为属性的，就不要抽象为实体
  • 厂家需要分配多的存储空间
  • 进行数据修改的时候需要考虑数据一致性

实体 or 属性

满足以下条件之一，则必须抽象成实体

  1. 含有 非 键 属性
  2. 事务处于“一对多” “多对多” 中多的那一端

否则，尽量抽象为属性
